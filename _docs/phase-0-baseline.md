# Phase 0 Baseline

## Scope

Phase 0 is documentation and contract groundwork only. The current behavior remains the compatibility baseline:

- Existing projects run as isolated local Supabase stacks.
- Provisioning still happens through the Supabase CLI.
- No new provisioning mode is introduced in this phase.

## Current Architecture

SupaConsole is a Next.js control plane that stores project metadata in PostgreSQL through Prisma and orchestrates local Supabase runtimes on demand.

Primary responsibilities are split as follows:

- `src/lib/project.ts`: lifecycle orchestration for initialize, create, deploy, pause, and delete.
- `src/lib/instances/*`: metadata model and persistence helpers for instance mode, topology, and runtime contract fields.
- `prisma/schema.prisma`: control-plane persistence model.
- `config.toml`: repository-level Supabase config template copied into each project workdir.
- `supabase-projects/<slug>`: per-project runtime workspace generated by SupaConsole.
- `supabase-core/`: checked-in reference code in this repository, but not a runtime dependency for project provisioning.

## Current Full-Stack Implementation

The only implemented provisioning path today is full-stack isolated local development.

### Initialize

`initializeSupabaseCore()` ensures these directories exist:

- `supabase-projects/`
- `supabase-core/`

It does not clone the Supabase monorepo. If `supabase-core/` is missing, it creates a placeholder README. In this repository, the embedded `supabase-core/` folder should be treated as read-only reference material.

### Create Project

`createProject()` currently resolves an instance profile but always defaults to:

- `provisioningMode = full_stack_isolated`
- `topologyKind = isolated_stack`
- `runtimeKind = supabase_cli_local`

Creation performs these steps:

1. Confirms the Supabase CLI is available.
2. Allocates a unique slug and a base port range.
3. Creates `supabase-projects/<slug>/`.
4. Writes generated runtime secrets and ports into:
   - `supabase-projects/<slug>/.env`
   - `supabase-projects/<slug>/docker/.env`
5. Persists the project record plus env vars in PostgreSQL.
6. Copies the repo-root `config.toml` into `supabase-projects/<slug>/supabase/config.toml`.

### Deploy Project

`deployProject()` is CLI-first, not template-compose-first:

1. Loads the project layout from stored topology metadata.
2. Runs Docker and connectivity pre-flight checks.
3. Re-assigns ports if conflicts are detected.
4. Runs `supabase start` in the project workdir.
5. Uses `supabase status -o json` to persist discovered URLs and runtime metadata.
6. Marks the project `active` if startup succeeds.

Docker Compose is used only as a fallback for stop/down flows or local container inspection, not as the primary provisioning entrypoint for project runtimes.

### Pause and Delete

- Pause uses `supabase stop --workdir .`, with `docker compose stop` as fallback.
- Delete uses `supabase stop --workdir .`, with `docker compose down --volumes --remove-orphans` as fallback, then removes the project directory and metadata records.

## Contract Fields Already Present

The database and metadata model already expose rollout-oriented fields:

- `Project.provisioningMode`
- `Project.topologyKind`
- `Project.runtimeKind`
- `Project.topologyMetadata`
- `Project.runtimeMetadata`
- `Project.secretMetadata`

The corresponding TypeScript enums also list future modes:

- `shared_core_schema_isolated`
- `db_isolated`
- `full_stack_isolated`

Those fields are groundwork only in Phase 0. They do not imply that shared-core or database-isolated provisioning exists today.

## Invariants To Preserve

The following are compatibility requirements for later phases:

- Existing projects created with `full_stack_isolated` must continue to work unchanged.
- `supabase start` remains the authoritative runtime start path for existing projects.
- Existing project workdirs remain under `supabase-projects/<slug>/`.
- Existing project env vars remain readable from both the database and the per-project `.env` file.
- Repo-root `config.toml` remains the source template copied into new project workdirs unless a deliberate migration replaces it.
- `initializeSupabaseCore()` must remain non-destructive and must not overwrite existing `supabase-projects/` or `supabase-core/` contents.
- The embedded `supabase-core/` tree remains read-only reference code, not mutable provisioning state.

## Known Gaps

These are real gaps in the current implementation and should inform later phases:

- The instance metadata contract already supports multiple modes, but the orchestration path still implements only `full_stack_isolated`.
- Secrets are still stored as inline env values, even though `secretMetadata` allows future indirection.
- Runtime status persistence is best-effort; failure to read status after a successful start still marks the project active.
- The root `docker-compose.yml` describes SupaConsole deployment, not the per-project runtime topology.
- Some setup/config documentation had drifted away from the live implementation before this Phase 0 pass.
- SMTP naming is not fully normalized across all config surfaces yet; later phases should standardize the app/runtime contract.

## Phase 1 Entry Criteria

Phase 1 should start from these assumptions:

- Full-stack isolated is the only supported provisioning mode.
- Any new mode rollout must be additive and explicitly opted into.
- Existing rows and project folders must not require automatic migration to continue working.
